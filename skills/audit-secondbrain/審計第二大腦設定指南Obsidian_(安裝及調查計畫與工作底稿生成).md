## 這個懶人包會幫你做什麼？

幫你從零開始建置審計專屬的 AI 第二大腦系統，一次完成所有步驟：

- ✅ 環境檢查與工具安裝（Node.js、Pandoc、Tesseract OCR、Python 轉換套件、Selenium）
- ✅ 安裝 mcpvault MCP Server（讓 Claude Code 能讀寫筆記）
- ✅ 批次檔案轉換腳本（docx、xlsx、pdf、圖片 → md，輸入 `/資料處理` 即可執行）
- ✅ 建好三層資料夾結構（資料蒐集 → 資料處理 → 資料庫建置及產出）
- ✅ 設定好 Claude 的工作規則（CLAUDE.md）
- ✅ 建好常用的筆記模板
- ✅ 設定每週自動知識重整排程
- ✅ 查核意見自動分類（依查核主題、建議事項、查明處理事項、注意事項等）
- ✅ 工作底稿生成規則（參考既有報告與生成原則產出標準化底稿）
- ✅ **法規 PCode 自動查詢工具**（get_pcode.py，使用 Selenium 模擬瀏覽器，確保法規連結正確）
- ✅ **調查計畫完整生成流程**（含相關法規及連結、調閱資料清單、缺失評估問卷，每一份均產出 MD + WORD 檔）

完成後，你只要做這些事：
1. **平時**：原始檔案（docx/pdf/圖片）放進 `資料蒐集/`

3. **生成調查計畫**：對 Claude 說「請幫我生成一份[主題]調查計畫」，AI 會自動產出調查計畫 + 法規連結 + 調閱清單 + 缺失問卷（MD + WORD）
4. **生成工作底稿**：對 Claude 說「請幫我根據[主題]生成工作底稿」
5. **每週日**：Claude 自動幫你整理成知識庫

## 階段一：環境檢查

### 步驟一：環境檢查

> 請 Claude 在開始前，先自動確認以下所有項目。
> 如果有任何一項不符合，請先告知使用者問題所在，並引導解決後再繼續。
> **不要跳過任何一項檢查，不要假設環境正常。**

1. **確認作業系統**：執行系統指令確認是 Windows / macOS / Linux
2. **確認網路連線正常**
3. **檢查 Node.js 是否已安裝**：
   - Windows 重要提醒：Claude Code 桌面版的 bash 環境可能找不到 `node`
   - 先嘗試 `node --version`
   - 若失敗，嘗試 `export PATH="/c/Program Files/nodejs:$PATH" && node --version`
   - 若仍失敗，代表未安裝，請安裝：
     - Windows：`winget install --id OpenJS.NodeJS --accept-source-agreements --accept-package-agreements`
     - macOS：`brew install node`
     - Linux：`sudo apt update && sudo apt install nodejs npm -y`
   - 安裝完成後，後續所有 node/npm/npx 指令都需要先加上 `export PATH="/c/Program Files/nodejs:$PATH"`（僅 Windows）
4. **檢查 npx 是否可用**：執行 `npx --version`（Windows 記得加 PATH）
5. **檢查 Pandoc 是否已安裝**（用於 docx 轉 md，xlsx 改用 pandas/openpyxl）：
   - 先嘗試 `pandoc --version`
   - 若失敗，嘗試以下路徑：
     - Windows（winget 安裝）：`%LOCALAPPDATA%\Microsoft\WinGet\Packages\JohnMacFarlane.Pandoc_Microsoft.Winget.Source_8wekyb3d8bbwe\pandoc-3.9.0.2\pandoc.exe --version`
   - 若仍失敗，請安裝：
     - Windows：`winget install --id JohnMacFarlane.Pandoc --accept-source-agreements --accept-package-agreements`
     - macOS：`brew install pandoc`
     - Linux：`sudo apt install pandoc -y`
   - 安裝後記錄 pandoc 的完整路徑（後續腳本會用到）
   - **xlsx 注意**：pandoc 對 xlsx 支援不穩定，xlsx → md 改使用 pandas + openpyxl（見步驟 7 套件安裝，自動覆蓋）
6. **檢查 Tesseract OCR 是否已安裝**（用於圖片文字辨識）：
   - 先嘗試 `tesseract --version`
   - 若失敗，檢查安裝路徑：`"C:\Program Files\Tesseract-OCR\tesseract.exe" --version`
   - 若仍失敗，請安裝：
     - Windows：`winget install --id UB-Mannheim.TesseractOCR --accept-source-agreements --accept-package-agreements`
     - macOS：`brew install tesseract`
     - Linux：`sudo apt install tesseract-ocr -y`
   - **安裝中文語系**（若無則下載）：
     ```bash
     # 繁體中文
     python -c "import urllib.request; urllib.request.urlretrieve('https://github.com/tesseract-ocr/tessdata_fast/raw/main/chi_tra.traineddata', r'C:\Program Files\Tesseract-OCR\tessdata\chi_tra.traineddata')"
     # 簡體中文
     python -c "import urllib.request; urllib.request.urlretrieve('https://github.com/tesseract-ocr/tessdata_fast/raw/main/chi_sim.traineddata', r'C:\Program Files\Tesseract-OCR\tessdata\chi_sim.traineddata')"
     ```
     macOS/Linux 下載至 `/usr/local/share/tessdata/` 或 `$(tesseract --print-parameters | grep tessdata_dir)`
7. **檢查 Python 轉換套件是否已安裝**：
   ```bash
   pip install pymupdf pytesseract pandas
   ```

8. **檢查 Selenium 及 webdriver-manager 是否已安裝**（用於法規 PCode 自動查詢驗證）：
   ```bash
   pip install selenium webdriver-manager
   ```
   > 💡 **為什麼需要 Selenium？** 全國法規資料庫會檢查 Session 和 Cookies，直接使用 requests 容易被擋。Selenium 可模擬瀏覽器行為，確保能正確搜尋法規名稱並取得 PCode，避免法規連結錯誤。

> 全部通過後，告知使用者環境狀態。

## 階段四：連接 MCP（讓 Claude Code 能讀寫筆記）

### 步驟六：安裝 mcpvault MCP Server

mcpvault 讓 Claude Code 能搜尋、讀取、編輯你的筆記。
**不需要 Obsidian 開著就能運作。不需要在 Obsidian 中安裝任何外掛。**

> 📝 **關於 mcpvault**：mcpvault 是一個輕量的 MCP 伺服器，可以直接讀寫 Obsidian vault 資料夾中的檔案，不需要透過 Obsidian 應用程式本身。

#### 6-1：全域安裝 mcpvault

```bash
# Windows（記得加 PATH）
export PATH="/c/Program Files/nodejs:$PATH" && npm install -g @bitbonsai/mcpvault

# macOS / Linux
npm install -g @bitbonsai/mcpvault
```

安裝完成後，確認全域安裝的路徑：
- Windows：通常在 `C:\Users\[使用者]\AppData\Roaming\npm\mcpvault.cmd`
- macOS / Linux：通常在 `/usr/local/bin/mcpvault` 或 `~/.npm-global/bin/mcpvault`

用 `which mcpvault` 或 `where.exe mcpvault` 確認實際路徑。

#### 6-2：寫入 MCP 設定檔

> ⚠️ **重要**：為了確保 Claude Code 能正確載入 MCP，請在**三個位置**都寫入設定。
> 這是因為不同版本的 Claude Code（桌面版、CLI、Web）讀取設定的位置不同。

**位置 1：使用者全域設定** `~/.claude/settings.json`
**位置 2：專案設定** `[工作目錄]/.claude/settings.local.json`
**位置 3：專案根目錄** `[工作目錄]/.mcp.json`

三個檔案內容相同：

```json
{
  "mcpServers": {
    "obsidian": {
      "command": "C:\\Users\\[使用者]\\AppData\\Roaming\\npm\\mcpvault.cmd",
      "args": [
        "[vault完整路徑]"
      ]
    }
  }
}
```

> 📝 **macOS / Linux 範例**：
> ```json
> {
>   "mcpServers": {
>     "obsidian": {
>       "command": "mcpvault",
>       "args": [
>         "/Users/[使用者]/secondbrain"
>       ]
>     }
>   }
> }
> ```

> ⚠️ 路徑中如果有中文或空格，在 JSON 中用雙反斜線跳脫。
> 例如：`"D:\\我的Vault\\secondbrain"`

### 步驟七：重啟 Claude Code 並驗證

> 🖐️ **需要手動操作**：請使用者完全關閉 Claude Code 桌面版，然後重新開啟。

重新開啟後，驗證 MCP 是否成功載入：
- 檢查工具清單中是否有 `mcp__obsidian__` 開頭的工具
- 若能回傳 vault 的資料夾結構，代表連接成功

如果 MCP 未成功載入，依序排查：
1. 確認 `~/.claude/settings.json` 中的 `command` 路徑是否正確
2. 確認 vault 路徑確實存在且可存取
3. 確認 `.mcp.json` 在 Claude Code 開啟的工作目錄下
4. 再次重啟 Claude Code

## 筆記庫結構（三層）

| 資料夾 | 用途 | 存什麼 |
|---|---|---|
| `資料蒐集/` | 原始資料 | docx、xlsx、pdf、圖片等原始檔案 |
| `資料處理/` | 轉換+分類 | 批次轉換後的 MD，依類型分至子資料夾 |
| `資料庫建置及產出/` | 產出文件 | 調查計畫、工作底稿、分析報告 |
| `工作日誌/` | 工作紀錄 | 每日工作紀錄、週計畫、知識重整週報 |
| `Templates/` | 模板 | 各種筆記的固定格式 |

## 資料處理子結構

資料處理/
├── 1.基本資料分析/     ← 分類：最近期→最舊時間 → 分析主題
├── 2.法規或函示/       ← 分類：主題 → 頒布時間(新→舊) → 頒布機關
└── 3.查核意見/         ← 分類：查核主題 → 建議事項/查明處理事項/注意事項 → 洞察/前瞻/監督

## 資料庫建置及產出子結構

資料庫建置及產出/
├── 調查計畫/           ← 依生成原則產出之調查計畫（含 MD + WORD）
├── 工作底稿/           ← 依生成原則產出之標準化底稿
└── 分析報告/           ← 綜合分析報告（無特定格式）

## 工作規則

### 新增筆記時
- 一律加上 frontmatter（title、date、tags）
- 根據內容判斷存到哪個資料夾：
  - 原始檔案（docx/xlsx/pdf/圖片）→ `資料蒐集/`
  - 轉換後的 MD → 一律放 `資料處理/`（依類型分至子資料夾），**不放** `資料庫建置及產出/`
    - 調查報告：完整報告→`1.基本資料分析/`，提取查核意見→`3.查核意見/`
    - 調查計畫、工作底稿原始稿、抽查報告等 → `1.基本資料分析/`
  - 每日/每週紀錄 → `工作日誌/`
- `資料庫建置及產出/` 僅放**依生成原則產出**之文件（調查計畫生成原則、工作底稿生成原則、爬蟲分析、二次分析等產出）

### 批次轉換規則
- 使用者輸入 `/資料處理` 時，執行以下完整流程（無需手動執行腳本）：
  1. **掃描**：僅掃描 `資料蒐集/` 目錄中所有 docx、xlsx、pdf、png、jpg 等原始檔案，**不掃描 vault 內其他目錄**
  2. **轉換**：將尚未轉換的原始檔案轉為 .md 格式：
     - docx → pandoc
     - xlsx → pandas + openpyxl（不經 pandoc，以確保表格結構穩定）
     - pdf → pymupdf
     - 圖片（png/jpg）→ Tesseract OCR
  3. **歸檔**：轉換後的 MD 檔直接存放至 `資料處理/` 下的對應子資料夾，**不可暫存於 `資料蒐集/`**
  4. **分類**：讀取每個 MD 檔內容進行類型判斷與歸檔：
     - **調查報告**（關鍵字：調查報告、查核報告、審計報告）：完整報告→`1.基本資料分析/`；提取「查核意見」章節→`3.查核意見/`
     - **法規或函示**（關鍵字：法規、條例、函、令）：→`2.法規或函示/`
     - **調查計畫、工作底稿原始稿、品質抽查報告**等：→`1.基本資料分析/`
     - **無法判斷類型者**：→`1.基本資料分析/`
  5. **提取查核意見**：若檔案類型為調查報告，自動掃描內容中「查核意見」章節（含 `#`、`##` 或 `**` 標題標記），將該章節內容獨立提取為一個 MD 檔存放至 `3.查核意見/`，並在每個意見末尾註記來源報告名稱
  6. **分類完成後**，自動執行重複性及冗餘資料清理：
     - 比對 `資料處理/` 下各子資料夾內 MD 檔的內容（跳過 frontmatter），刪除內容完全相同之重複檔案
     - 若同一份調查報告因多次提取產生多個相似檔案，保留檔名最完整或內容最完善者，其餘刪除
     - 清理後輸出摘要報告（含刪除檔案清冊、保留檔案清冊、各資料夾筆數統計）
- **所有轉換後的 MD 檔僅存放於 `資料處理/` 下，不進入 `資料庫建置及產出/`**
- **清理完成後**，更新 `查核意見總整理` 與 `查核意見明細`（存入 `3.查核意見/`，檔名以民國曆日期後綴）
- **最後**，詢問使用者：「分類與清理已完成。是否刪除 `資料蒐集/` 中的所有原始檔案（含已轉換的原始檔及暫存檔）？刪除後可釋放磁碟空間，但無法恢復原始檔案。」等待使用者確認後才執行刪除

### 轉換後檔案分類規則

#### 0. 檔案類型判斷（分類前自動判斷）
自動掃描 MD 檔內容（frontmatter + 首段），依以下關鍵字判斷檔案類型：

| 類型 | 判斷關鍵字 | 歸檔目錄 |
|------|-----------|---------|
| **調查報告** | 調查報告、查核報告、審計報告、專案調查 | `1.基本資料分析/`（完整）+ `3.查核意見/`（意見章節） |
| **法規或函示** | 法、條例、規則、辦法、函、令、公告、解釋 | `2.法規或函示/` |
| **調查計畫** | 調查計畫、查核計畫、調查規劃 | `1.基本資料分析/` |
| **工作底稿原始稿** | 工作底稿、底稿、查核紀錄 | `1.基本資料分析/` |
| **品質抽查報告** | 抽查、抽核、品質查核 | `1.基本資料分析/` |
| **無法判斷** | 無明顯關鍵字 | `1.基本資料分析/` |

#### 1. 基本資料分析分類規則
- 主要分類：依時間由最近期至最舊排列
- 次分類：依分析主題歸類

#### 2. 法規或函示分類規則
- 主要分類：依主題歸類
- 次分類：依頒布時間由新至舊排序
- 再次分類：依頒布或函示機關歸類
- 每則法規/函示需註明來源文件名稱

#### 3. 查核意見分類規則（依序）
- **第一層**：查核主題（主要分類）
- **第二層**：建議事項、查明處理事項、注意事項（次分類）
- **第三層**：洞察、前瞻、監督（再次分類）
- **第四層**：查核意見標題及其文章內容
- 每個意見最後需註記截取自哪一份調查報告的名稱
- 分類時先讀取調查報告內容，提取「查核意見」章節進行分類

### 工作底稿生成規則
- **角色設定**：應以「資深審計人員」角度進行生成
- **參考來源**：
  1. `資料蒐集/` 中的查核資料
  2. `資料處理/3.查核意見/` 中相關案例
- **生成結構（四階段）**：
  1. **標題**：粗體、200字內。首段摘要查核標的項目、中段摘要查核事實內容，格式：「機關+事項+缺失摘要+允待/允宜/允應...，以...。」尾段需呼應擬議處理意見
  2. **依據**：引用行為時之法令/規定/函文，全段文字截錄（不摘要），依查核事實內容順序對應擇選，注意事件發生時間點以截取行為時之法令
  3. **查核事實**：架構為「機關主體 + 查核標的(規模/金額) + 執行情形/現況(含大事記) + 缺失分析(惟查...核有...疏失/核欠妥適)」
  4. **擬議處理意見**：200字內，與標題尾段呼應，可用「綜上/鑑於」開頭結合查核事實推論總結

### 調查計畫生成規則

#### 生成格式
- **資料來源**：網路爬蟲 + `資料蒐集/` + `資料處理/` 中所有相關資料

調查計畫格式及內容包含以下章節：

**調查計畫名稱**（置頂，格式：「○○縣市政府辦理○○○執行情形調查計畫」）

1. **壹、計畫緣起**：說明調查背景、目的與動機，字數至少400~600字（可引永續發展目標或上級政策背景）
2. **貳、調查依據**：制式格式為「依據審計法第12條、第59條辦理。」
3. **參、調查規劃方法**：制式格式為「運用『規劃矩陣』(Planning Matrix)，研提本專案調查之研究問題、評量標準及資訊之來源、查核範圍及方法、查核限制、可能提出之查核意見等(詳如附件)，俾利調查計畫之撰擬，提升查核品質。」
4. **肆、調查對象及範圍**
   - 一、調查對象
   - 二、調查範圍
5. **伍、調查重點**：至少5項以上（若涉及跨單位合作，可加註協辦單位）
6. **陸、調查方式**
   - 一、本案先函請相關單位查填調查表並據以辦理研析
   - 二、擬採書面審核或就地稽察方式辦理，調查結束後，將調查結果及擬議處理意見依限函送相關單位彙整，相關電子檔案寄送至本部相關單位承辦人員電子郵件信箱
7. **柒、調查時應行注意事項**：臚列調查前中後應注意事項（參考既有調查計畫範例樣式撰寫，至少10項以上）
8. **捌、調查人員責任**：載明領組及調查人員之職責（參考既有調查計畫範例樣式撰寫，至少4項以上）
9. **玖、調查人員及調查天數**：制式格式為「為兼顧調查人員對調查工作之瞭解，並充分運用現有人力，經審視各員之專業知能、技術、經驗及能力，擬由本室第三課審計○○以就地稽察或書面調查方式辦理，調查報告預計於__年__月__日前提出。」
10. **拾、調查所需費用**：由使用者自訂查核期間及天數，調查所需差旅費計算式為查核天數×400元+查核天數×交通費100元
11. **附件：專案調查計畫規劃矩陣**（表格格式，共5欄位，依序為：研究問題（編號1~5）、評量標準及資訊之來源（合併欄，內含「評量標準」及「資訊之來源」子標題）、查核範圍及方法（包括資料可靠性）（合併欄，內含「查核範圍」及「查核方法」子標題）、查核限制、可能提出之查核意見（編號1~5）。MD 檔以 markdown 表格呈現，WORD 檔以 Word 表格呈現）

#### 生成後續作業（需詢問使用者意願）

**重要**：調查計畫生成完成後（同時產出 MD 檔及 WORD 檔），**必須先詢問使用者是否要接續製作以下三份配套文件**。不可未經詢問直接自動生成。使用者可選擇：
- 全部製作
- 僅製作其中部分（例如只做調閱資料清單）
- 跳過不製作

若使用者選擇製作，則依據生成的調查計畫內容（包括計畫緣起、調查對象及範圍、調查重點及附件：專案調查計畫規劃矩陣等），進行網路爬蟲與法規搜尋，並分析各調查重點查核所需資料及評估可能會發生的缺失，生成以下三份文件（每份均同時產出 MD 檔及 WORD 檔）：

1. **相關法規及連結**：依調查重點整理相關法規條文（全文顯示，不可摘要或截取）及來源連結（指向法規全文頁面，格式為 `https://law.moj.gov.tw/LawClass/LawAll.aspx?pcode=XXXX`）。生成後**必須驗證**：
   - 使用 `get_pcode.py` 或以瀏覽器實際開啟確認 pcode 對應法規名稱正確
   - 確認網址可正常開啟且顯示正確法規（非404、非導向其他法規）
   - **不可跳過驗證步驟**
2. **調閱資料清單**：依調查計畫內容（調查主題、調查對象及範圍、調查重點）分析被調查機關應提供之資料，動態生成清單項目（非固定十二項，項目數及內容依個案而定），格式如下：
   - **開頭說明**：「審計部臺灣省宜蘭縣審計室為瞭解貴府及所屬（貴公所）辦理○○情形，預計於○○前往貴機關辦理查核。敬請安排小開會議室(查核人員○員)。並請就下列議題分別準備各該案(表1)相關資料(電子檔優先，若尚未辦理或無資料，請說明後免提供)依序排列：」
   - **資料清單項目**（依調查重點動態產生，例如工程案件可能含：補助計畫及核定函、預算編列及支用情形、委託規劃設計監造、工程招決標變更設計、工程契約書、品管制度、監造紀錄、承包商自主品管、估驗驗收結算、工程進度報表、財產維護管理與使用情形等；營運管理案件則需對應列出營運績效評定、契約執行情形等不同項目）
   - **表1 抽查案件明細**（僅當抽查案件數大於1件時才須列出）：欄位含項次、標案名稱及件數、主辦機關、決標金額，並列示合計金額

### 步驟九：建立法規 PCode 查詢工具（get_pcode.py）

> 💡 **這個工具為什麼重要？** 生成「相關法規及連結」時，法規的 PCode 必須正確（例如促參法 pcode=D0070062，而非 D0070013）。全國法規資料庫會檢查 Session 和 Cookies，直接用 requests 抓取容易被擋，故需使用 Selenium 模擬瀏覽器查詢。

在 vault 根目錄建立 `get_pcode.py`，內容如下：

```python
"""
全國法規資料庫 PCode 查詢工具
使用 Selenium 模擬瀏覽器，搜尋法規名稱並取得正確 PCode

用法：
    python get_pcode.py "法規名稱"

範例：
    python get_pcode.py "促進民間參與公共建設法"
    python get_pcode.py "公路法"
    python get_pcode.py "汽車運輸業管理規則"

輸出：
    法規名稱、PCode、完整網址（法規全文頁面）

注意：
    需先安裝 Selenium 及瀏覽器驅動：
    pip install selenium webdriver-manager
"""

import sys
import time
import re
from selenium import webdriver
from selenium.webdriver.common.by import By
from selenium.webdriver.common.keys import Keys
from selenium.webdriver.chrome.service import Service
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC
from webdriver_manager.chrome import ChromeDriverManager


def search_law_pcode(law_name: str) -> dict | None:
    """
    在全國法規資料庫搜尋法規名稱，回傳正確的 PCode

    流程：
    1. 開啟全國法規資料庫首頁
    2. 在搜尋框輸入法規名稱並搜尋
    3. 從搜尋結果中找到正確法規條目
    4. 擷取 PCode（排除含 Log 參數的暫時性網址）
    5. 回傳結果
    """
    options = webdriver.ChromeOptions()
    options.add_argument("--headless=new")
    options.add_argument("--no-sandbox")
    options.add_argument("--disable-dev-shm-usage")
    options.add_argument("--disable-gpu")
    options.add_argument("--window-size=1920,1080")
    options.add_argument(
        "user-agent=Mozilla/5.0 (Windows NT 10.0; Win64; x64) "
        "AppleWebKit/537.36 (KHTML, like Gecko) "
        "Chrome/120.0.0.0 Safari/537.36"
    )

    driver = webdriver.Chrome(
        service=Service(ChromeDriverManager().install()),
        options=options,
    )

    try:
        print(f"正在搜尋：「{law_name}」...")
        driver.get("https://law.moj.gov.tw/")
        wait = WebDriverWait(driver, 15)

        # 找到搜尋框並輸入法規名稱
        search_box = wait.until(
            EC.presence_of_element_located(
                (By.CSS_SELECTOR, "input[type='text']")
            )
        )
        search_box.clear()
        search_box.send_keys(law_name)
        search_box.send_keys(Keys.RETURN)
        time.sleep(3)

        # 從搜尋結果中找出正確法規的連結（排除含 Log 參數的暫時性網址）
        all_links = driver.find_elements(By.TAG_NAME, "a")
        target_url = None

        for link in all_links:
            href = link.get_attribute("href")
            if not href:
                continue
            if "Log=true" in href or "&Log=" in href:
                continue
            if "law.moj.gov.tw/LawClass/" not in href:
                continue
            pcode_match = re.search(r'pcode=([A-Z]\d+)', href)
            if pcode_match:
                pcode = pcode_match.group(1)
                target_url = f"https://law.moj.gov.tw/LawClass/LawAll.aspx?pcode={pcode}"
                break

        if target_url:
            result = {
                "name": law_name,
                "pcode": re.search(r'pcode=([A-Z]\d+)', target_url).group(1),
                "url": target_url,
            }
            print(f"  ✅ 找到：{result['name']} → PCode：{result['pcode']}")
            return result
        else:
            print(f"  ❌ 找不到「{law_name}」的 PCode")
            return None

    except Exception as e:
        print(f"  ❌ 搜尋「{law_name}」時發生錯誤：{e}")
        return None
    finally:
        driver.quit()


def batch_search(law_names: list[str]) -> list[dict]:
    """批次搜尋多部法規"""
    results = []
    for name in law_names:
        result = search_law_pcode(name)
        if result:
            results.append(result)
        print()
    return results


if __name__ == "__main__":
    if len(sys.argv) > 1:
        law_name = " ".join(sys.argv[1:])
        search_law_pcode(law_name)
    else:
        print("=" * 60)
        print("全國法規資料庫 PCode 批次查詢")
        print("=" * 60)
        law_list = [
            "審計法", "促進民間參與公共建設法", "公路法",
            "汽車運輸業管理規則", "身心障礙者權益保障法",
            "建築技術規則建築設計施工編", "空氣污染防制法",
            "停車場法", "民法", "政府採購法",
        ]
        results = batch_search(law_list)
        print("=" * 60)
        print("查詢結果總表")
        print("=" * 60)
        for r in results:
            print(f"{r['name']:<30} {r['pcode']:<15} {r['url']}")
```

> 建立完成後，告知使用者可以先測試：「執行 `python get_pcode.py "促進民間參與公共建設法"` 確認是否能正確取得 PCode。」

## 階段十二：批次轉換既有檔案（docx、xlsx、pdf、圖片 → md）

### 步驟十七：了解批次轉換流程

當使用者在 `資料蒐集/` 資料夾放置非 MD 檔案（docx、xlsx、pdf、圖片等）時，
只需輸入 `/資料處理`，Claude 就會自動執行以下流程：

1. **掃描**：僅掃描 `資料蒐集/` 中的原始檔案（docx、xlsx、pdf、png、jpg），**不掃描 vault 內其他目錄**
2. **轉換**：依格式使用對應工具轉為 MD：
   - docx → pandoc
   - xlsx → pandas + openpyxl（不經 pandoc，確保表格結構穩定）
   - pdf → pymupdf
   - 圖片 → Tesseract OCR（chi_tra + chi_sim）
3. **歸檔**：轉換後的 MD 直接存放至 `資料處理/` 下的對應子資料夾，**不可暫存於 `資料蒐集/`**
4. **類型判斷**：讀取每個 MD 檔內容，依關鍵字判斷檔案類型：
   - **調查報告**（關鍵字：調查報告、查核報告、審計報告）：完整報告→`1.基本資料分析/`；提取「查核意見」章節→`3.查核意見/`
   - **法規或函示**（關鍵字：法、條例、函、令）：→`2.法規或函示/`
   - **調查計畫、工作底稿原始稿、品質抽查報告**等：→`1.基本資料分析/`
   - **無法判斷**：→`1.基本資料分析/`
5. **分類排列**：依時間（最近→最舊）排列
6. **提取查核意見**：若為調查報告，自動掃描「查核意見」章節（`#`、`##`、`**` 標題），獨立提取為 MD 檔存放至 `3.查核意見/`，每個意見末尾註記來源報告名稱
7. **原則**：所有轉換後的 MD 僅存放於 `資料處理/`，**不進入** `資料庫建置及產出/`（後者僅放依生成原則產出之文件）
8. **清理**：分類完成後，自動比對內容刪除重複檔案（跳過 frontmatter，全文比對）及冗餘資料（多次提取之相似檔案僅保留最完整者），並輸出清理摘要報告（含刪除清冊、保留清冊、各資料夾筆數統計）
9. **更新查核意見總整理與查核意見明細**：清理後自動重新掃描 `3.查核意見/`，提取分類、去重、產生日期版索引檔
10. **詢問**：清理完成後，詢問使用者：「分類與清理已完成。是否刪除 `資料蒐集/` 中的所有原始檔案？」等待使用者確認後才執行刪除
8. **更新查核意見總整理（含去重）**：清理完成後，自動掃描 `資料處理/3.查核意見/` 中所有 MD 檔案，提取其中所有查核意見（依 `**一、**` `**二、**` 等中文編號粗體標題辨識），執行以下步驟：
   - **提取**：掃描每個 MD 檔，找出所有中編號粗體標題（`**一、...**`、`**二、...**` 等），將標題後至下一個標題或檔尾的內容作為該意見的本文
   - **去重**：對每筆意見的本文全文計算 MD5 雜湊值，僅保留首次出現者，移除內容完全相同之重複條目（跨檔案比對）
   - **分類**：依標題關鍵字自動判斷主題類型（工程品質、採購程序、財務管理等）及意見分類（查明處理事項/建議事項/注意事項）
   - **產生**兩份文件，以當日日期（民國年月日，例如 `1150520`）作為檔名後綴：
     - `查核意見總整理(當日日期).md` — 摘要統計表（含主題類型、分類、年度、筆數、去重紀錄等）
     - `查核意見明細(當日日期).md` — 逐條完整版（含編號、主題類型、分類、出處、年度、標題、內容、附圖附表索引）
   - 若檔案已存在則直接覆蓋，確保兩份文件始終反映 `3.查核意見/` 最新內容

### 查核意見分類規則（詳細）

**第一層（主要分類）**：依查核主題區分
- 例如：工程品質、採購程序、財務管理、內部控制等

**第二層（次分類）**：依意見類型區分
- 建議事項
- 查明處理事項
- 注意事項

**第三層**：依性質區分
- 洞察（發現潛在問題）
- 前瞻（預防性建議）
- 監督（追蹤改善）

**第四層**：查核意見標題及文章內容
- 直接摘錄查核意見原始內容
- 每個意見末尾註記：「（截取自：[調查報告名稱]）」

### 查核意見章節提取方法

當檔案被判斷為**調查報告**時，自動提取「查核意見」章節：

1. **定位章節**：掃描 MD 內容，尋找標題中包含「查核意見」的行（支援 `# 查核意見`、`## 查核意見`、`**查核意見**` 等格式）
2. **提取範圍**：從「查核意見」標題下一行開始，至下一個同層級或更高層級標題為止（或檔尾）
3. **分割意見項目**：將提取出的內容依 `**一、**`、`**二、**` 等中編號粗體標題逐項分割，每項獨立為一個查核意見
4. **註記來源**：在每個分割出的意見末尾附加「（截取自：[報告名稱]）」
5. **存放**：獨立存為一個 MD 檔（檔名格式為 `[報告名稱]_查核意見.md`），放入 `3.查核意見/`

### 查核意見總整理與查核意見明細（自動維護）

每次執行 `/資料處理` 並完成分類清理後，系統會自動：

1. **掃描** `資料處理/3.查核意見/` 下所有 MD 檔案（排除 `查核意見總整理` 及 `查核意見明細` 本身）
2. **提取**每個檔案中的查核意見：
   - 標題：以 `**` 包圍之中文編號粗體文字（如 `**一、...**`、`**二、...**`），須排除非意見之粗體文字（表數字、附錄章節、壹貳參等章節編號）
   - 內容：標題所在行後至下一個標題行或檔尾之全文
   - 出處：從 frontmatter 或檔案原始名稱取得
   - 年度：從檔名 `[XXX]` 標記取得
   - 附圖附表：從內容中 `[詳附件` 索引取得
3. **去重**：對每筆意見的**本文全文**計算 MD5 雜湊值，建立已見過之 MD5 集合；僅保留首次出現之條目，後續相同 MD5 者全部跳過。去重**跨所有檔案**（同一意見若同時出現在「完整報告」與「提取版」檔案中，僅保留首次出現的一份）
4. **分類**：依標題關鍵字自動判斷主題類型（如：工程品質、採購程序、財務管理、轉運站管理等）及意見分類（查明處理事項/建議事項/注意事項）
5. **產生**兩份文件，存放於 `資料處理/3.查核意見/`（**非** `分析報告/`）：
   - `查核意見總整理(YYYYMMDD).md` — 摘要表格，依主題類型統計各年度筆數，含去重紀錄
   - `查核意見明細(YYYYMMDD).md` — 逐條詳列，含完整欄位與全文
   - 檔名後綴為民國曆日期（如 `1150520`），確保每次更新均可追溯版本

**注意**：此兩份文件為 `3.查核意見/` 的索引總表，須與來源檔案放置同一目錄，**不可**移回 `分析報告/`，以避免批次轉換時漏掃。

### 轉換腳本（進階用）

以下 Python 腳本為批次轉換的底層實作，
一般使用者只需輸入 `/資料處理` 即可，不需手動執行腳本。

```python
"""轉換腳本存放於 vault 根目錄：convert_all_to_md.py"""
# 設定區
VAULT_PATH = r"[你的vault路徑]"
PANDOC_PATH = r"[pandoc完整路徑]"
TESSERACT_PATH = r"[tesseract完整路徑]"
# ...（腳本完整內容略，詳見 vault 根目錄的 convert_all_to_md.py）
```

#### 轉換方式對照表

| 原始格式 | 轉換工具 | 說明 |
|---------|---------|------|
| docx | pandoc | 穩定支援格式與樣式 |
| xlsx | pandas + openpyxl | 逐工作表轉為 MD 表格，保留欄位結構（pandoc 對 xlsx 支援不穩，故改用此方案） |
| pdf | pymupdf（fitz） | 提取純文字與表格 |
| 圖片（png/jpg） | Tesseract OCR（chi_tra+chi_sim） | 辨識繁體/簡體中文 |

> **xlsx 轉換說明**：pandas + openpyxl 會將每個工作表轉為獨立的 markdown 表格區塊，並以工作表名稱作為區塊標題。此法可保留表格欄位名稱與資料列，遠比 pandoc 的 xlsx 轉換穩定。

### 轉換後檔案管理

- **原始檔案保留**：轉換不會自動刪除原始 docx/xlsx/pdf/圖片，僅在旁邊產生暫存 MD 檔。待分類完成至 `資料處理/` 後，經使用者同意可一併刪除 `資料蒐集/` 中所有檔案
- **手動新增檔案**：以後有新檔案，直接放到對應資料夾，再說「執行轉換腳本」即可
- **選擇性轉換**：可指定單一檔案，說「幫我把 XXX.docx 轉成 md」
- **查核意見索引檔**：`資料處理/3.查核意見/` 下的 `查核意見總整理(YYMMDD).md` 與 `查核意見明細(YYMMDD).md` 由系統自動維護，每次 `/資料處理` 執行後會自動更新。**請勿手動編輯**此兩份檔案，手動新增查核意見應直接放入新 MD 檔至該目錄，下次執行 `/資料處理` 時會自動納入。

## 常見問題

| 問題 | 解法 |
|------|------|
| mcpvault 搜尋不到筆記 | 確認 vault 路徑正確，路徑中有中文或空格需用引號包住 |
| `npx: command not found` | 確認 Node.js 已安裝，重啟 Claude Code 桌面版 |
| 重啟後 MCP 工具仍不存在 | 確認設定檔中 `command` 使用**完整路徑**（如 `C:\\Users\\...\\mcpvault.cmd`） |
| Windows bash 找不到 node | `export PATH="/c/Program Files/nodejs:$PATH"` |
| 設定檔寫了但沒生效 | 確認在三處都寫入（`~/.claude/settings.json`、`.claude/settings.local.json`、`.mcp.json`） |
| Obsidian 需要裝外掛嗎？ | **不需要**。mcpvault 直接讀寫 vault 資料夾的檔案，不經過 Obsidian app |
| pandoc 找不到 | 用完整路徑執行，或加入 PATH 環境變數 |
| PDF 轉 md 後格式亂掉 | PDF 非結構化格式，pymupdf 僅提取純文字。若需保留表格可用 pandas 輔助 |
| OCR 辨識率不佳 | 確認圖片解析度夠高，可先試用 Claude Code 的 Vision 功能直接讀圖 |
| **新檔案加入後如何轉換？** | 直接放進對應資料夾，對 Claude 說「執行轉換腳本」即可 |
| **生成法規連結時 pcode 錯誤怎麼辦？** | 使用 `python get_pcode.py "法規名稱"` 查詢正確 pcode，確認後將網址改為 `https://law.moj.gov.tw/LawClass/LawAll.aspx?pcode=XXXX` 格式 |
| **Selenium 啟動時報錯 ChromeDriver 問題** | 自動下載失敗時，手動至 https://googlechromelabs.github.io/chrome-for-testing/ 下載對應版本，或執行 `pip install --upgrade webdriver-manager` |
| **全國法規資料庫搜尋不到法規** | 確認法規名稱正確（使用完整名稱如「促進民間參與公共建設法」而非簡稱），若仍搜尋不到可改用手動開啟 https://law.moj.gov.tw 查詢 |
